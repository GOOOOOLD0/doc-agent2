# Doc Agent Repository Instructions

## 1. 项目定位

本项目以 Hermes Agent 作为统一入口，通过 Skill 路由和 Python 工具维护两类知识：

1. 各国卫星落地许可与市场准入；
2. 书籍、法规、标准及其他长文档的解析与 Source Notes。

知识库位于：

`wiki/`

本文件只定义仓库级导航、Skill 路由和全局约束。具体业务流程由对应的 `SKILL.md`、配置文件和模板负责。

---

## 2. Skill 路由

### Satellite Landing Rights

Skill：

`skills/satellite-landing-rights/SKILL.md`

适用于：

- 查询、研究、核验或比较某国卫星落地许可；
- 分析市场准入、服务授权、频率、设备认证、地面站、网关站和用户终端许可；
- 收集官方来源并生成 Source Notes；
- 创建或更新国家落地许可案例。

### Book Ingestion

Skill：

`skills/book-ingestion/SKILL.md`

适用于：

- 将书籍、法规、标准、报告或其他长文档加入 Wiki；
- 解析 PDF、DOCX 等原始文件；
- 识别章、条、节、款等结构；
- 生成或检查 `parsed/`、`chunks/` 和 `source_notes/`；
- 更新书籍 Source Notes 索引。

### 路由规则

1. 执行任务前，先选择最相关的一个 Skill。
2. 用户目标与国家卫星落地许可相关时，加载 `satellite-landing-rights`。
3. 用户目标与长文档解析、切分或 Source Notes 相关时，加载 `book-ingestion`。
4. 同时涉及文档入库和落地许可分析时，先执行 `book-ingestion`，再执行 `satellite-landing-rights`。
5. Git、Linux、Python、普通代码修改和一般知识问答通常不加载业务 Skill。
6. 不要调用另一个完整 Agent 处理已由 Hermes 接管的任务；优先调用确定性的项目工具。

---

## 3. 仓库结构

- `skills/`：业务流程、触发条件和执行规则；
- `tools/`：解析、切分、校验、索引和安全写入等确定性能力；
- `config/`：文档类型、切分方式和工具参数；
- `wiki/raw/`：原始来源、解析结果、Chunks 和 Source Notes；
- `wiki/concepts/`：基于充分证据形成的正式知识；
- `wiki/comparisons/`：跨国家或跨主题的对比结果。

不得把未经证据支持的正式结论写入 `wiki/raw/`，也不得把原始文档或中间解析结果写入 `wiki/concepts/`。

---

## 4. 指令优先级

项目内部规则发生冲突时，按以下顺序处理：

1. 用户当前明确要求；
2. 本文件中的全局约束；
3. 当前加载的 `SKILL.md`；
4. 对应的 YAML 配置、common 规则和模板；
5. 历史案例、旧输出和示例文件。

仅在规则不冲突时采用优先级较低的内容。发现冲突时，应说明冲突，不得静默忽略。

实际存在的文件、当前配置和正式模板优先于旧示例。

---

## 5. 全局约束

1. 重要结论必须可追溯到 Source Note、Chunk 或官方来源。
2. 不得把推测、模型常识或其他国家经验写成已确认事实。
3. 来源冲突、证据不足或解析异常时，明确标记为待确认。
4. 不得擅自修改原文含义、条款编号、历史版本号或修订标记。
5. 原始文件应保留，不得被清洗结果、摘要或生成文件覆盖。
6. 用户只要求问答时，不修改 Wiki。
7. 仅在用户明确要求创建、保存或更新时写入文件。
8. 自动生成的正式内容默认保持 `draft` 或待复核状态。
9. 不覆盖已人工确认的内容，除非用户明确要求。
10. 不自动执行 Git commit、push、删除文件或发送外部信息。
11. 不生成只有标题、没有证据的空文件。
12. 工具失败时保留已有结果，说明失败阶段，不进行破坏性重跑。

---

## 6. 标准执行流程

1. 检查当前目录和实际存在的文件；
2. 根据用户意图加载对应 Skill；
3. 读取完成任务所需的最少上下文；
4. 优先调用已有工具，不重复实现相同功能；
5. 生成结果后执行最小验证；
6. 向用户报告结果、依据和待复核事项。

不要无条件读取整个 Wiki。应先读取索引，再下钻到相关页面、Source Notes、Chunks 或原始来源。

---

## 7. 最小验证

写入或更新文件后至少检查：

1. 文件位于正确目录；
2. 文件名、一级标题和 frontmatter 符合对应规则；
3. 引用的 Source Note、Chunk 和内部链接实际存在；
4. 页码、版本号、条款编号和来源信息未被错误修改；
5. 关键结论具有来源；
6. 未产生意外空文件、重复文件或未完成占位内容；
7. 相关索引已同步更新。

无法完成某项检查时，应明确说明，不得默认验证通过。

---

## 8. 模块职责

Hermes Agent 负责：

- 理解用户意图；
- 选择 Skill；
- 规划步骤；
- 调用文件、网页和项目工具；
- 汇总结果并与用户交互。

Skill 负责：

- 业务边界；
- 工作流程；
- 证据和写入规则；
- 任务完成条件。

Python 工具负责：

- 可重复的确定性操作；
- 输入输出校验；
- 解析、切分、索引和格式检查；
- 预览与安全写入。

详细规则应放在对应模块中，不要继续复制到本文件。

---

## 9. 回答风格

1. 默认使用中文，除非用户要求其他语言；
2. 先给结论，再说明必要步骤；
3. 操作类任务给出明确路径、文件名和命令；
4. 避免一次提供过多无关选项；
5. 不确定内容必须明确说明；
6. 完成写入后，列出创建或更新的文件以及待复核事项。

---

## 10. 运行约定

1. `AGENTS.md`、`skills/`、`tools/` 和 `wiki/` 位于项目根目录；
2. 尽量从项目根目录启动 Hermes；
3. 项目只保留一个主 `AGENTS.md`；
4. 新增业务能力时，优先新增或修改对应 Skill；
5. 新增工具参数和文件格式时，优先更新对应配置或模块说明，不扩充根文件。
